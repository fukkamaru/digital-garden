Mibなどの単位表現はいつ頃から登場したのか？ スマホなどが広まって1024ではなく、≒1000表記が当たり前になったのがキッカケですか？

ざっくり結論だけ先に言うと、MiBなどはスマホ普及よりかなり前に作られた表記です。背景は「kilo/mega/gigaを1000にも1024にも使っていて紛らわしい問題」の整理です。公式情報ベースで年表を確認します。

Thought for 4m 56s

登場時期は **1998〜1999年ごろ**です。  
正確には、IECが **1998年12月に承認**し、**1999年1月発行の IEC 60027-2 Amendment 2** で、KiB・MiB・GiB などの2進接頭辞が標準化されました。NISTもその経緯を説明しています。

ポイントは、**スマホ普及がキッカケではありません**。  
むしろ原因はもっと古くからある、

> KB / MB / GB が、  
> 1000倍なのか、1024倍なのか分からない問題

です。

## まず表記の整理

|表記|意味|バイト数|
|---|---|---|
|KB|kilobyte|1,000 B|
|KiB|kibibyte|1,024 B|
|MB|megabyte|1,000,000 B|
|MiB|mebibyte|1,048,576 B|
|GB|gigabyte|1,000,000,000 B|
|GiB|gibibyte|1,073,741,824 B|

NISTの例でも、**1 MiB = 2²⁰ B = 1,048,576 B**、**1 MB = 10⁶ B = 1,000,000 B** と分けられています。

なお、細かいですが **MiB** と **Mib** は違います。

|表記|意味|
|---|---|
|MiB|mebibyte。バイト|
|Mib|mebibit。ビット|

大文字 **B = byte**、小文字 **b = bit** です。

## なぜMiBなどが作られたのか

昔のコンピュータでは、2進数の都合で **1024 = 2¹⁰** が扱いやすかった。  
しかも1024は1000に近いので、現場ではざっくり **1KB = 1024B** のように使われました。

でも本来、SI接頭辞の kilo / mega / giga は **10の累乗**です。NISTのSIガイドでも、kiloを1024、megaを1,048,576、gigaを1,073,741,824の意味で使うのは不適切だと明記されています。

そこで、

- 1000倍系：KB / MB / GB
- 1024倍系：KiB / MiB / GiB

に分けましょう、という整理が生まれました。

## スマホ普及が原因ではない

スマホ普及は**原因というより、一般ユーザーがこのズレに気づきやすくなった要因**です。

この問題自体は、スマホ以前からHDD・メモリ・通信速度などで存在していました。NISTも、ストレージ機器メーカーはMBを **1,000,000バイト** の意味で使うことが多く、コンピュータメモリでは **2²⁰ = 1,048,576バイト** の意味で使われることが多かったため、混乱が生じたと説明しています。

つまり流れとしてはこうです。

1. 昔のコンピュータ業界では、KB/MB/GBを1024倍系で使うことが多かった
2. HDDやSSDなどのストレージメーカーは、GBを1000倍系で表示していた
3. OSやソフトによっては1024倍系で容量を表示した
4. 「500GBのHDDなのに、PCでは465GBくらいに見える」問題が起きた
5. その混乱を避けるため、MiB/GiBなどの2進専用表記が作られた

## 「≒1000表記が当たり前になった」というより

ここは少し言い方を分けた方がいいです。

**GBやMBは、本来は“約1000”ではなく、10進法で正確に1000倍です。**  
つまり、

- 1 KB = 1,000 B
- 1 MB = 1,000,000 B
- 1 GB = 1,000,000,000 B

です。

一方で、

- 1 KiB = 1,024 B
- 1 MiB = 1,048,576 B
- 1 GiB = 1,073,741,824 B

です。

なので「1000表記が当たり前になった」というより、**本来のSI接頭辞の意味に戻して、1024倍系には別名を与えた**という理解が近いです。

## ただし普及はかなり中途半端

MiB / GiB は標準化されていますが、一般向け表示では今でも **MB / GB** が多いです。  
たとえばMacは、OS X 10.6 Snow Leopard以降、ディスク容量表示を10進法寄りにしたと説明されています。 一方でWindowsなどは、内部的には1024倍で計算しているのに表示はGBと書く場面があり、ここが混乱の温床です。

## まとめると

MiBなどは、**スマホ時代に生まれた表記ではなく、1998〜1999年に標準化された“1024倍専用の単位表現”**です。

キッカケはスマホではなく、昔からあった **「MB/GBが1000倍なのか1024倍なのか曖昧」問題**。  
スマホやSSDの普及は、そのズレを一般ユーザーにも見えやすくした、という位置づけです。

2010年前後に大学で情報工学を学んでいたときに耳にしなかったので、そのような表記や呼び名があることをgit関係を調べていて知りました。

Thought for 11s

それはかなり自然です。  
**2010年前後に情報工学を学んでいても、KiB / MiB / GiB をほぼ聞かなかった人は普通に多い**と思います。

理由は、標準化された時期と、教育・現場で普及した時期がかなりズレているからです。

## 標準化は早いが、現場では浸透が遅かった

KiB / MiB / GiB などの2進接頭辞は、IECによって1998〜1999年に標準化されています。NISTも、2進接頭辞は IEC 60027-2 Amendment 2 で採用されたものだと説明しています。

ただし、標準化されたからといって、大学の講義・教科書・OS表示・開発現場で一斉に使われたわけではありません。

2010年前後なら、授業ではまだ普通に、

- 1KB = 1024B
- 1MB = 1024KB
- 1GB = 1024MB

のように教わることが多かったはずです。

厳密にはこれは **KiB / MiB / GiB** の内容なのですが、当時の説明では **KB / MB / GB** のまま済ませていた、という感じです。

## Gitで知ったのは納得感があります

Gitでは、clone / fetch / push などの進行表示で、

```
Receiving objects: ..., 10.18 MiB | 7.63 MiB/sWriting objects: ..., 1.14 MiB | 1.88 MiB/s
```

のように **MiB / MiB/s** が出てきます。GitLabの公式ドキュメント例でも `81.06 MiB | 5.74 MiB/s` のような表記が使われています。

なので、GitやLinux系ツールを触っていて初めて「あれ、MBじゃなくてMiBって何？」となるのはかなり“あるある”です。

## なぜ大学で出てこなかったのか

理由はたぶんこのあたりです。

|場面|よく使われた表記|
|---|---|
|アルゴリズム・プログラミング|あまり容量単位を厳密に扱わない|
|コンピュータアーキテクチャ|KB / MB を1024倍の意味で使いがち|
|OS・メモリ管理|page size 4KB、cache 32KB など、KB表記が慣習|
|HDD・SSD・USBメモリ|GBは1000倍系で表示されがち|
|Linux/Git系ツール|KiB / MiB / GiB が比較的出やすい|

つまり、**情報工学の中でも、MiB表記が必要になる場面はわりと限定的**です。

特に授業では「2進数だから1024倍」という概念が大事で、表記の厳密性までは重視されにくい。  
だから先生も教科書も「本当はMiBだけど、ここではMBと書く」みたいな運用になりがちです。

## 2010年前後の感覚としては

かなり雑に言うと、当時はまだこういう空気だったと思います。

> 厳密にはMiBだけど、  
> みんなMBって書いてるし、文脈で分かるよね。

で、GitやLinux系のツールは比較的きっちり **MiB** と出す。  
一方で、Windowsや一般向け製品表示では **GB / MB** が普通に使われ続ける。

このズレが残ったまま今に至っています。

## なので理解としては

かなり正確に言うなら、

> MiBなどは1999年ごろから存在したが、  
> 2010年前後の大学教育や一般的なIT利用では、まだ広く目にする表記ではなかった。  
> Gitなどの開発ツールを触る中で知るのは自然。

という感じです。

ちなみに、**MiBは「ミビバイト」**, **GiBは「ギビバイト」** ですが、口頭で使うと少しクセが強いので、現場でも「メビ」「ギビ」と軽く言う人もいれば、そもそも口に出さず表記だけ使う人も多いです。

では2010年にgitを使っていた、linuxをバリバリ使っていたら、当然のようにMibやKibといった単語表記は普通に目にしていたのか？それとも、それでも目にする機会は少なかったのか？

この問いは「標準化されたか」ではなく「2010年当時の実体験として目にしたか」ですね。Gitの進捗表示、Linux系コマンド、デスクトップ環境の表示を分けて確認します。

Thought for 59s

結論としては、**GitをCLIでそこそこ使っていたなら、2010年時点でも MiB / KiB を目にする機会は普通にありました。**  
ただし、**Linuxをバリバリ使っていたら当然のように日常的に見る、というほどではなかった**と思います。

まず表記は細かくいうと、容量なら **MiB / KiB** です。  
**Mib / Kib** は bit 側、つまり mebibit / kibibit になります。

## Gitでは2010年でも見えていた可能性が高い

Git 1.7.0 は2010年2月に公開されています。  
その Git 1.7.0 の `progress.c` には、進捗表示用に **GiB / MiB / KiB / MiB/s / KiB/s** を出す処理がすでに入っています。

つまり、2010年にCLIで、

```
git clonegit fetchgit pullgit push
```

などをしていて、ある程度サイズのあるリポジトリを扱っていれば、

```
Receiving objects: ..., 12.34 MiB | 1.23 MiB/sWriting objects: ..., 456.78 KiB | ...
```

のような表示を見ていた可能性はあります。

ただし、これは**進捗表示の横に一瞬出る単位**です。  
なので、見ていたとしても「そういう単語として認識していなかった」可能性はかなりあります。Gitの操作中って、普通はパーセントやエラーしか見ないので。

## Linux全般では、むしろ少なめ

一方で、Linuxの基本コマンドでは事情が違います。

たとえば GNU coreutils の昔からの説明では、`human-readable` 表示は1024倍系でも、表示は **M** や **K** のような短い文字を使う、と説明されています。`M` は 1,048,576 bytes、`MB` は 1,000,000 bytes、`1M` と `1MiB` は同じ、という扱いです。

つまり、

```
ls -lhdu -shdf -h
```

を使っても、よく見るのはだいたい、

```
4.0K128M2.3G
```

であって、**4.0 KiB / 128 MiB / 2.3 GiB** とは表示されにくいです。

なので、Linuxをよく使っていても、日常的に `MiB` を浴びるように見るかというと、そこまでではありません。

## ただしLinux界隈の配布サイトなどでは出ていた

一方で、GNOMEの2010年前後のソース配布ディレクトリを見ると、ファイルサイズに **MiB / KiB** が普通に使われています。たとえば GNOME 2.30.2 のソース一覧では、`1.9 MiB`、`497.4 KiB`、`2.3 MiB` のような表記が並んでいます。

なので、2010年時点でもLinux/OSS界隈に存在感はありました。  
ただ、「OSの授業や普段のコマンドで必ず目に入る表記」ではなく、**Gitの進捗表示、ソース配布サイト、厳密なドキュメント、設定値の説明などで見る表記**だった、という感じです。

## 感覚としてはこうです

|2010年ごろの場面|MiB / KiBを目にする可能性|
|---|---|
|Gitでclone/fetch/pushをよくする|高め|
|Gitを小さい課題・ローカル中心で使う|低め|
|Linuxで`ls -lh` / `du -sh` / `df -h`を使う|低め|
|Linuxの配布サイト・OSSソース一覧を見る|中くらい|
|OS・情報工学の大学講義|低め|
|厳密な仕様書・マニュアルを読む|中くらい|

なので、答えはこうです。

**2010年にGitをCLIでしっかり使っていた人なら、MiB / KiBを見ていた可能性は十分ある。**  
でも、**Linuxをバリバリ使っていても、普通のコマンド操作だけなら意外と見ない。**

あなたが大学時代に聞かなかったのは不自然ではないです。  
当時の教育・会話では、まだ普通に「1MBは1024×1024バイト」みたいに言って、表記だけはMBのまま済ませる文化が強かったはずです。GitやOSS配布サイト側の方が、大学講義より先に厳密表記を出していた、くらいの理解がしっくりきます。
