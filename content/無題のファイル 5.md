ANAMEやCNAMEとは何か？BNAMEや他の名称はありますか？また、ドメインや、サブドメイン、カスタムドメインといった用語とも合わせて教えてほしい。

思考時間: 2 ～ 3 秒

DNSまわりの用語ですね。ざっくり言うと、**ドメイン名をどこへ向けるかを決める仕組み**です。

まず全体像から言うと、

```
example.com├─ www.example.com├─ blog.example.com└─ shop.example.com
```

このうち、

- `example.com` が **ドメイン**
- `www.example.com` や `blog.example.com` が **サブドメイン**
- 自分で持っているドメインを外部サービスに接続して使うものが **カスタムドメイン**

です。

たとえば、GitHub Pages や Cloudflare Pages や Notion、Shopify などに、  
`fukkamaru.space` や `notes.fukkamaru.space` を割り当てる場合、それが **カスタムドメイン設定**です。

---

## DNSレコードとは何か

DNSレコードは、簡単に言うと、

> この名前にアクセスされたら、どこへ案内するか

を決める設定です。

代表的には以下です。

|種類|役割|例|
|---|---|---|
|A|ドメインをIPv4アドレスへ向ける|`example.com → 192.0.2.1`|
|AAAA|ドメインをIPv6アドレスへ向ける|`example.com → 2001:db8::1`|
|CNAME|ある名前を別のドメイン名の別名にする|`www.example.com → example.pages.dev`|
|MX|メール配送先を指定する|Gmailなど|
|TXT|認証情報・所有確認などを書く|SPF, DKIM, Google認証|
|NS|そのドメインのDNS管理先を指定する|Cloudflareなど|
|SRV|特定サービスの接続先を指定する|通話・チャット系など|
|CAA|SSL証明書を発行できる認証局を制限する|Let’s Encryptのみ許可など|

---

## Aレコードとは

**Aレコード**は、ドメイン名をIPv4アドレスに向ける設定です。

```
example.com → 192.0.2.1
```

つまり、

> `example.com` に来た人を、このサーバーのIPアドレスへ案内する

という設定です。

昔ながらの一番基本的なDNSレコードです。

---

## AAAAレコードとは

**AAAAレコード**は、AレコードのIPv6版です。

```
example.com → 2001:db8::1
```

AがIPv4、AAAAがIPv6です。

名前が「AAAA」なのは少し変ですが、IPv6アドレスがIPv4より大きいので、Aの4倍というニュアンスでAAAAになっています。

---

## CNAMEとは

**CNAME**は「Canonical Name」の略で、あるドメイン名を別のドメイン名の別名にする設定です。

たとえば、

```
www.example.com → example.pages.dev
```

と設定すると、

> `www.example.com` は `example.pages.dev` の別名です

という意味になります。

Cloudflare Pages や GitHub Pages などでよく使います。

### CNAMEの特徴

CNAMEはIPアドレスではなく、**別のドメイン名へ向ける**のがポイントです。

```
OK:www.example.com → example.pages.devNG:www.example.com → 192.0.2.1
```

IPアドレスへ直接向けるならAレコード、  
別のドメイン名へ向けるならCNAMEです。

---

## CNAMEの注意点

CNAMEは便利ですが、重要な制約があります。

通常、**ルートドメインにはCNAMEを設定できません**。

```
example.com        ← ルートドメインwww.example.com    ← サブドメイン
```

多くの場合、CNAMEは `www.example.com` のようなサブドメインには使えますが、  
`example.com` のようなルートドメインには使いにくいです。

理由は、ルートドメインにはNSレコードやSOAレコードなど、DNS管理上必須のレコードが存在するためです。CNAMEは「この名前は完全に別名です」という扱いなので、他のレコードと同居できないのが基本です。

---

## ANAMEとは

**ANAME**は、CNAMEに似た仕組みです。

特に、

> ルートドメインでもCNAMEっぽいことをしたい

という目的で使われます。

たとえば、本来ならルートドメインにはCNAMEを置きづらいですが、

```
example.com → example.pages.dev
```

のようなことをしたい場合があります。

このときに使われるのが、DNSサービス独自の **ANAME** や **ALIAS** です。

ただし重要なのは、**ANAMEは標準的なDNSレコードではありません**。

DNSサービス会社が独自に用意している便利機能です。

---

## ALIASとは

**ALIAS**もANAMEとほぼ同じ目的で使われます。

つまり、

> ルートドメインでCNAMEのように別ドメインへ向けたい

ときの仕組みです。

サービスによって名前が違います。

|名称|よくある意味|
|---|---|
|ANAME|CNAME風に別ドメインへ向ける独自機能|
|ALIAS|ANAMEとほぼ同系統の独自機能|
|Flattened CNAME|CNAMEを裏側でA/AAAAに変換する機能|
|CNAME Flattening|Cloudflareなどで使われる表現|

要するに、だいたい同じ方向性です。

---

## BNAMEはあるのか

**BNAME**という名前は、一般的なDNS設定ではほぼ見かけません。

A、AAAA、CNAME、MX、TXT、NSなどは一般的ですが、  
BNAMEは少なくとも通常のWebサイト公開やカスタムドメイン設定では使わないと思ってよいです。

「AがあるならBもあるのでは？」と思いがちですが、DNSはアルファベット順に名前が付いているわけではありません。

---

## よく出てくる名称の整理

### ドメイン

インターネット上の住所のようなものです。

```
fukkamaru.spaceexample.comgoogle.com
```

など。

---

### ルートドメイン / Apexドメイン / Nakedドメイン

サブドメインが付いていない本体部分です。

```
example.comfukkamaru.space
```

これを、

- ルートドメイン
- Apexドメイン
- Nakedドメイン
- Zone Apex

などと呼ぶことがあります。

厳密には少し文脈差がありますが、実務上はだいたい、

> `www` などが付いていない素のドメイン

という理解で大丈夫です。

---

### サブドメイン

ドメインの前に名前を付けたものです。

```
www.example.comblog.example.comshop.example.comnotes.example.com
```

`www`、`blog`、`shop`、`notes` の部分がサブドメインです。

---

### カスタムドメイン

外部サービスが用意したURLではなく、自分のドメインを割り当てることです。

たとえば Cloudflare Pages なら、最初はこういうURLになります。

```
example.pages.dev
```

そこに自分のドメインを割り当てて、

```
www.example.comexample.comnotes.example.com
```

などでアクセスできるようにする。

これがカスタムドメインです。

---

## 実務でよくある設定例

### 例1：`www.example.com` を外部サービスに向ける

```
www.example.com CNAME example.pages.dev
```

これは分かりやすいです。

`www.example.com` は `example.pages.dev` の別名、という設定です。

---

### 例2：`example.com` を外部サービスに向ける

ルートドメインなので、通常のCNAMEは使いづらいです。

この場合は、DNSサービス側で以下のような機能を使います。

```
example.com ALIAS example.pages.dev
```

または、

```
example.com ANAME example.pages.dev
```

または Cloudflare なら、

```
example.com CNAME example.pages.dev
```

としても、裏側で **CNAME Flattening** により処理してくれることがあります。

---

## A / CNAME / ANAME・ALIASの違い

|種類|向ける先|主な用途|ルートドメインで使えるか|
|---|---|---|---|
|A|IPv4アドレス|自前サーバーや固定IP|使える|
|AAAA|IPv6アドレス|IPv6対応サーバー|使える|
|CNAME|別のドメイン名|サブドメインを外部サービスへ接続|原則ルートでは不可|
|ANAME|別のドメイン名風|ルートでCNAMEっぽく使う|サービスによる|
|ALIAS|別のドメイン名風|ルートでCNAMEっぽく使う|サービスによる|

---

## かなり実務寄りに言うと

Webサイト公開で迷ったら、だいたいこうです。

|やりたいこと|使うことが多い設定|
|---|---|
|`www.example.com` を外部サービスに向ける|CNAME|
|`blog.example.com` を外部サービスに向ける|CNAME|
|`example.com` を固定IPのサーバーに向ける|A / AAAA|
|`example.com` を外部サービスに向ける|ANAME / ALIAS / CNAME Flattening|
|メールを使う|MX / TXT|
|ドメイン所有確認|TXT|
|SSL証明書まわりの制限|CAA|

---

## まとめ

一番大事なのはここです。

```
A       = ドメイン名 → IPアドレスAAAA    = ドメイン名 → IPv6アドレスCNAME   = ドメイン名 → 別のドメイン名ANAME   = ルートドメインでもCNAMEっぽく使う独自機能ALIAS   = ANAMEと似た独自機能
```

そして、

```
example.com      = ドメイン、ルートドメインwww.example.com  = サブドメイン自分のドメインを外部サービスに割り当てる = カスタムドメイン
```

という理解でOKです。

実務では、**サブドメインならCNAME、ルートドメインならA/AAAAまたはALIAS/ANAME/CNAME Flattening** と覚えておくとかなり整理しやすいです。

思考時間: 1 秒

はい、その理解で**大筋OK**です。  
ただし、正確に言うと **CNAME自体がAレコードのように動くわけではなく、「最終的にA/AAAAレコードを探しに行く」** という挙動です。

流れはこうです。

```
www.example.com にアクセス↓DNSを見る↓www.example.com は CNAME で example.pages.dev の別名だと分かる↓では example.pages.dev のIPアドレスは？↓example.pages.dev の A / AAAA レコードを調べる↓IPアドレスが返る↓そのサーバーへ接続する↓サーバー側が www.example.com 用のページを返す
```

つまり、

```
www.example.com CNAME example.pages.devexample.pages.dev A     192.0.2.10
```

のような関係です。

なので、実際には、

> `www.example.com` が直接IPアドレスを持っているのではなく、  
> `example.pages.dev` の向き先を借りて、結果的にそのサーバーへ行く

という感じです。

---

ただし、ここで大事なのがもう一つあります。

DNSで目的のサーバーに到達しただけでは、**どのページを表示するかまでは完全には決まりません**。

サーバー側、たとえば Cloudflare Pages や GitHub Pages 側で、

```
www.example.com はこのサイトに対応しています
```

という設定が必要です。

つまりカスタムドメイン設定では、だいたい次の2つがセットです。

```
DNS側:www.example.com → example.pages.devサービス側:www.example.com はこのPagesプロジェクトに割り当てる
```

DNSは「どのサーバーへ行くか」を決める。  
サービス側のカスタムドメイン設定は「そのドメインで来たアクセスに、どのサイトを返すか」を決める。

ここを分けて考えると、かなりスッキリします。

たとえるなら、

```
DNS = 建物まで案内するサーバー側の設定 = 建物内のどの部屋に通すか決める
```

です。

はい、かなり近いです。  
ただし、より正確には **「ConoHa側で見る」「Cloudflare側で見る」というより、“そのドメインのDNS管理者が誰か”を順番にたどる** という理解がよいです。

あなたの想定がこうだとします。

```
自分のドメイン:fukkamaru.spaceDNS管理:ConoHa接続先:Cloudflare Pages の example.pages.dev
```

この場合の流れはこうです。

```
① ブラウザで www.fukkamaru.space にアクセス↓② DNSリゾルバが fukkamaru.space のDNS管理先を確認↓③ fukkamaru.space のDNSを管理しているConoHaに問い合わせる↓④ ConoHa側のDNS設定を見る   www.fukkamaru.space CNAME example.pages.dev↓⑤ 次に example.pages.dev の名前解決へ進む↓⑥ pages.dev はCloudflareが管理しているので、Cloudflare側のDNSを見る↓⑦ example.pages.dev の A / AAAA レコード相当のIPアドレスが返る↓⑧ そのIPアドレスへ接続↓⑨ Cloudflare Pages側が www.fukkamaru.space 用のサイトを返す
```

なので、あなたの言い方に寄せると、

```

```

という理解でOKです。

ただし注意点として、実際にDNSを見に行くのはブラウザそのものというより、通常は **DNSリゾルバ** です。たとえば、プロバイダのDNS、Google DNS、Cloudflare DNS、ルーター、OSのDNSキャッシュなどが関わります。

ざっくり役割で分けるとこうです。

|役割|何をするか|
|---|---|
|ブラウザ|`www.fukkamaru.space` にアクセスしたい|
|DNSリゾルバ|その名前のIPアドレスを調べる|
|ConoHa DNS|`www.fukkamaru.space` は `example.pages.dev` だと答える|
|Cloudflare DNS|`example.pages.dev` のIPアドレスを答える|
|Cloudflare Pages|実際のWebサイトを返す|

ポイントは、**ConoHaがCloudflareのIPを知っているわけではない**ことです。

ConoHaはあくまで、

```
www.fukkamaru.space は example.pages.dev の別名です
```

と答えるだけ。

その先の、

```
example.pages.dev はどのIPなのか？
```

は、`pages.dev` を管理しているCloudflare側に聞きに行く、という流れです。

私の言い方に合わせる必要はありません。私の理解があっているのか？間違っているのかが重要です。 設定はそれぞれ各サーバー（conoHa, cloudflare）で行ったが、該当URL（ドメイン）にアクセスしたときの挙動では、裏側ではサーバーは関係ないのか？ 別のシステム上で処理されているのか？ということを聞きたいのです。

思考時間: 1 秒

重要な点だけ言うと、あなたの理解は**半分合っていて、半分ズレています**。

結論はこれです。

> ドメインにアクセスしたとき、最初に動くのは ConoHa や Cloudflare Pages の「Webサーバー」ではなく、DNSの仕組みです。  
> ただし、そのDNS情報を持っている場所として、ConoHaやCloudflareのDNSサーバーは関係します。

つまり、**Webサーバーとは別の仕組み**です。

---

## 「サーバー」と言ったときに2種類ある

ここが混乱ポイントです。

|種類|役割|
|---|---|
|DNSサーバー|ドメイン名から接続先を調べる|
|Webサーバー|実際のWebページを返す|

ConoHaやCloudflareという会社名で見ると同じ「サーバー」に見えますが、実際には役割が違います。

---

## あなたのケースで分けると

たとえば、

```
www.fukkamaru.space↓ CNAMExxxxx.pages.dev
```

のように設定している場合。

### 1. 最初に関係するのはDNSサーバー

この段階では、まだWebページにはアクセスしていません。

```
www.fukkamaru.space の接続先はどこ？
```

をDNSで調べています。

このとき、`fukkamaru.space` のDNS管理がConoHaなら、**ConoHaのDNSサーバー**が関係します。

ただしこれは、ConoHaのWebサーバーではありません。

---

### 2. CNAMEの先をさらにDNSで調べる

ConoHaのDNSが、

```
www.fukkamaru.space は xxxxx.pages.dev の別名です
```

と返す。

すると今度は、

```
xxxxx.pages.dev のIPアドレスはどこ？
```

を調べます。

`pages.dev` はCloudflare側の管理なので、ここでは **CloudflareのDNSサーバー**が関係します。

これもまだ、Cloudflare PagesのWebページを返す処理ではありません。

---

### 3. IPアドレスが分かってからWebサーバーへ接続する

DNSでIPアドレスが分かったあと、ようやくWebアクセスです。

```
このIPアドレスのサーバーに接続して、www.fukkamaru.space のページをください
```

となります。

ここで初めて、**Cloudflare Pages側のWeb配信サーバー**が関係します。

---

## つまり、流れはこう

```
ブラウザ↓DNSリゾルバ↓ConoHaのDNSサーバー  「www.fukkamaru.space は xxxxx.pages.dev です」↓CloudflareのDNSサーバー  「xxxxx.pages.dev のIPはこれです」↓Cloudflare PagesのWebサーバー  「www.fukkamaru.space 用のページを返します」
```

---

## なので質問への答え

> 設定はそれぞれ各サーバーで行ったが、該当URLにアクセスしたときの挙動では、裏側ではサーバーは関係ないのか？

**関係あります。**  
ただし、最初に関係するのはWebサーバーではなく、**DNSサーバー**です。

> 別のシステム上で処理されているのか？

**はい。DNS名前解決という別の仕組みで処理されています。**

そのあとに、実際のページ表示としてWebサーバーへアクセスします。

## 一番正確な理解

こう覚えるのがよいです。

```
DNS設定を見る段階= WebサーバーではなくDNSサーバーの仕事ページを表示する段階= Webサーバーの仕事
```

そして、

```
ConoHaでDNS管理している= ConoHaのDNSサーバーが答えるCloudflare Pagesでサイトを公開している= CloudflareのWeb配信サーバーがページを返す
```

です。

なので、**「ConoHaのサーバーにアクセスしてからCloudflareへ飛ばしている」わけではありません。**

ここがかなり大事です。

ConoHaのDNSは、あくまで住所案内をしているだけ。  
実際のWebアクセス先は、最終的に名前解決されたCloudflare側のIPアドレスです。

理解した。扱っている2つのDNSサーバーで処理をして、その結果の該当するWEBサーバーへ接続したということですね。

はい、その理解でOKです。

少しだけ厳密に言うと、

```
① DNSリゾルバがDNSサーバーに問い合わせる② ConoHa側のDNSサーバーが CNAME を返す③ Cloudflare側のDNSサーバーが CNAME先のIPアドレスを返す④ そのIPアドレスのWebサーバーへ接続する⑤ Webサーバーが該当ドメイン用のページを返す
```

という流れです。

なので、かなり正確にはこうです。

> 2つのDNSサーバーが「名前解決」に関わり、  
> その結果として得られたIPアドレスのWebサーバーへ接続する。

ここで大事なのは、**ConoHaのWebサーバーを経由してCloudflareへ転送しているわけではない**ことです。  
ConoHa側はDNSとして「その名前はCloudflare Pages側の名前ですよ」と案内しているだけです。

たとえるなら、

```
ConoHa DNS：「www.fukkamaru.space は xxxxx.pages.dev を見てください」Cloudflare DNS：「xxxxx.pages.dev の住所はこのIPです」Cloudflare Pages：「そのドメインで来たなら、このサイトを表示します」
```

という感じです。

cloudflare pagesやgithub pagesも、そのサービス側でドメインを取得・管理していたら、カスタムドメイン利用でもAレコードやAAAAで接続出来るのか？

DNSの一般論と、Cloudflare Pages / GitHub Pages の実務設定は少し分けて確認します。特に GitHub Pages は公式が指定IPを持っているので、そこは確実に押さえます。

思考時間: 4s

はい。**サービス側がそのドメインのDNSを管理しているなら、カスタムドメインでもA/AAAAで接続できる場合があります。**  
ただし、これは **サービスごとの設計次第** です。

## まず一般論

DNS管理者がどこであれ、A/AAAAレコードはこういう設定です。

```
example.com A    192.0.2.10example.com AAAA 2001:db8::10
```

つまり、

> このドメイン名は、このIPアドレスへ向ける

という設定です。

なので、CloudflareやGitHub側でドメインを取得・管理している場合でも、**そのDNS管理画面でA/AAAAを設定できるなら接続自体は可能**です。

ただし大事なのは、**接続先サービスがA/AAAA用の固定IPを公開しているか**です。

---

## GitHub Pagesの場合

GitHub Pagesは、apexドメイン、つまり `example.com` のようなルートドメイン向けに、AレコードとAAAAレコード用のIPアドレスを公式に案内しています。公式ドキュメントでは、Aレコードは GitHub Pages のIPv4アドレスへ、AAAAレコードはIPv6アドレスへ向ける、と説明されています。

つまり GitHub Pages は、

```
example.com → A / AAAA → GitHub PagesのIP
```

という接続ができます。


## なぜA/AAAAだけではダメなのか

A/AAAAは、接続先を**IPアドレスで固定**します。

- 障害時の迂回

- セキュリティ対策
- CDN側の構成変更
- IPv4 / IPv6 / HTTP/3 などネットワーク側の改善
- サービス全体のインフラ移行

このとき、ユーザーがA/AAAAで固定IPを入れていると、**IPを変えるたびにユーザー全員へDNS設定変更を求める必要**があります。これはかなり面倒です。

違います。かなり違います。

|方式|ユーザー側|サービス側|
|---|---|---|
|A/AAAA固定IP|分かりやすい|IPを長期間維持する必要がある|


GitHub Pagesは、ルートドメイン用にA/AAAAのIPを公式に出しています。公式ドキュメントでも、apexドメインにはGitHub PagesのAレコード用IPv4、AAAAレコード用IPv6を設定できると案内されています。

CloudflareはそもそもDNS/CDN/プロキシ基盤を持っている会社なので、

```

```

A/AAAA固定IP方式は、

> GitHub Pagesはルートドメイン用にA/AAAAレコードで指定できるIPアドレスを公式に案内している。一方、Cloudflare PagesはユーザーがDNS設定に使うための固定IPを公開しておらず、Cloudflare DNS上のCNAME Flatteningによってルートドメインを接続する仕組みになっている。

この表現なら、かなり誤解が少ないです。

Cloudflare DNS上のCNAME Flatteningが他所で管理しているドメインには使えないという公式説明はどこにある？

公式説明の該当箇所を確認して、Cloudflare Pages側の「apexはCloudflare DNS必須」と、DNS側の「CNAME FlatteningはCloudflare DNSの機能」の2点に分けて示します。

思考時間: 4s

公式説明として一番直接的なのは、Cloudflare Pages の **Custom domains** です。

Cloudflareは、外部DNSを使う場合について、

> Cloudflareにネームサーバーを向けたくない場合は、Cloudflare Pagesで使うには**サブドメイン用のカスタムCNAMEレコードを作る必要がある**

と説明しています。例も `shop.example.com → <YOUR_SITE>.pages.dev` です。つまり、外部DNS管理では案内されているのが **サブドメインのCNAME** で、ルートドメインではありません。

一方、Cloudflareの **CNAME flattening** の説明では、

> CNAME flattening は `example.com` のような **zone apex** にCNAMEレコードを使えるようにする  
> そして、この機能が Cloudflare Pages の **root custom domain** を使えるようにしている

と説明されています。

さらに設定ページでは、

> CNAME flattening は、あなたのドメインが **zone apexにCNAMEレコードを使っているとき** にデフォルトで発生する

と説明されています。つまり、これは **Cloudflare DNS上のゾーンにあるCNAMEレコードをCloudflareがflattenする機能**です。

なので、正確にはこうです。

```
外部DNS管理のドメイン→ CloudflareのDNSゾーン上にapex CNAMEを置けない→ Cloudflare DNSのCNAME Flatteningも使えない→ Cloudflare PagesではサブドメインCNAMEを使う案内になる
```

補強として、Cloudflareの別ドキュメントでは、CNAME setupではRFC 1912に従いCNAMEはzone apexに置けないため、zone apexを扱うには**権威DNSプロバイダがCNAME flattening相当をサポートしている必要がある**、とも説明されています。これは「Cloudflare管理でないなら、その外部DNS側がflattening/ALIAS/ANAME相当を持っていないと無理」という意味です。

結論としては、

> **Cloudflare Pagesのルートドメイン利用は、Cloudflare DNSのCNAME Flatteningに依存する。  
> 外部DNS管理ではCloudflare側のCNAME Flatteningをそのドメインに適用できないため、公式案内はサブドメインCNAMEになる。**

という理解でOKです。

では、github pagesでは、他所でドメイン管理していた場合はCNAMEを使うわけですが、それをcname flatteningみたいに、ルートドメインにすることはできますか？

GitHub Pagesの場合は「GitHub側のCNAME Flattening」ではなく、外部DNS側の機能か、GitHub公式IPへのA/AAAAで処理します。公式の言い方に寄せて整理します。

思考時間: 4s

はい、**できます**。  
ただし、Cloudflare Pagesのように「GitHub側のCNAME Flattening」を使うのではなく、**ドメインを管理しているDNSサービス側の機能**で実現します。

GitHub Pages公式では、ルートドメイン、つまり `example.com` のような apex domain は、次のどちらかで設定すると案内されています。

```
example.com A     GitHub PagesのIPv4アドレスexample.com AAAA  GitHub PagesのIPv6アドレス
```

または、

```
example.com ALIAS / ANAME username.github.io
```

GitHub公式ドキュメントでも、apexドメインは `A`、`ALIAS`、または `ANAME` レコードで構成すると説明されています。サブドメインは `CNAME` で構成します。

つまり、外部DNS管理の場合でも、GitHub Pagesのルートドメインは可能です。

## 方式は2つ

### 1. A / AAAAレコードでGitHub Pages公式IPへ向ける

これが一番標準的です。

```
example.com A    185.199.108.153example.com A    185.199.109.153example.com A    185.199.110.153example.com A    185.199.111.153
```

必要ならAAAAも設定します。GitHub公式は、apexドメインを構成するには、表にあるすべての `A` / `AAAA` レコードを追加するか、代わりに `ALIAS` / `ANAME` を追加すると説明しています。

### 2. DNS事業者側のALIAS / ANAME / CNAME flattening相当を使う

DNS管理サービスが対応しているなら、

```
example.com ALIAS username.github.io
```

または、

```
example.com ANAME username.github.io
```

のように設定できます。

これは実質的に、

> ルートドメインでCNAMEっぽいことをする

ための仕組みです。

ただし、この処理をするのは**GitHubではなく、あなたが使っているDNSプロバイダー側**です。

## CNAMEそのものをルートドメインに置くわけではない

ここが大事です。

```
example.com CNAME username.github.io
```

を普通のCNAMEとして置くのは、DNS仕様上かなり問題があります。  
ルートドメインにはNSやSOAなどの必須レコードがあるため、通常のCNAMEとは同居できません。

なので、GitHub Pagesでルートドメインを使う場合は、

```
A / AAAA
```

または、

```
ALIAS / ANAME / DNSプロバイダー側のCNAME flattening相当
```

です。

## 結論

> GitHub Pagesを他所のDNSで管理していても、ルートドメイン利用はできます。  
> 方法は、GitHub公式IPへのA/AAAA、または外部DNS側が対応しているALIAS/ANAME/CNAME flattening相当です。

Cloudflare Pagesとの違いはここです。

|項目|GitHub Pages|Cloudflare Pages|
|---|---|---|
|外部DNSでルートドメイン|可能|基本的に不可・公式案内はサブドメインCNAME|
|主な方法|A/AAAA または ALIAS/ANAME|Cloudflare DNS上のCNAME Flattening|
|固定IPの案内|あり|ユーザー向け固定IPは案内しない|
|CNAME flatteningを誰がやるか|外部DNS事業者側|Cloudflare DNS側|

なので、GitHub Pagesはこの点ではかなり融通がききます。

GitHub Pagesは、**ユーザーがDNS設定に使える公式IPアドレスを公開している**ので、外部DNS管理でもルートドメインを使えます。

```
example.com A     GitHub PagesのIPv4example.com AAAA  GitHub PagesのIPv6
```

という形ですね。

つまり、

> ドメイン管理がConoHaやお名前.comやXserver側にあっても、  
> そのDNS管理画面でGitHub Pages公式のA/AAAAレコードを設定すれば、  
> `example.com` のようなルートドメインをGitHub Pagesへ向けられる。

ということです。

Cloudflare Pagesとの違いはここです。

```
GitHub Pages→ A/AAAA用の公式IPを公開している→ 外部DNSでもルートドメインを向けられるCloudflare Pages→ ユーザー向け固定IPを公開していない→ ルートドメインはCloudflare DNS側のCNAME Flatteningで処理する→ 外部DNSでは基本的にサブドメインCNAMEになる
```

なので、今回の整理としては、

> **GitHub Pagesは固定IP方式を許している。  
> Cloudflare PagesはCNAME方式に寄せて、ルートドメインはCloudflare DNS側で吸収している。**

この理解でかなりきれいです。
