# Obsidian / Quartz における改行思想と Zettelkasten 運用についての議論整理

## 1. 発端：「Semantic Line Breaks」のサイト探し

最初の話題は、

- `sentence-per-line`
- `semantic line breaks`

のような、Markdown における改行思想について。

ユーザーは：

- 英語で文章を書いている
- 日本の「守破離」について本を書いている
- 文書の改行について論じている

人物のサイトを探していた。

その結果、該当サイトとして：

- [Writing Slowly](https://writingslowly.com/?utm_source=chatgpt.com)

が見つかった。

特に：

- [Semantic line breaks are a feature of Markdown, not a bug](https://writingslowly.com/2025/01/13/semantic-line-breaks-are-a.html?utm_source=chatgpt.com)

の記事が対象。

---

# 2. Semantic Line Breaks とは何か

議論では、Semantic Line Breaks の核心として：

> 「改行を見た目ではなく意味単位で行う」

という思想を整理した。

これは：

- sentence-per-line（1文1行）
- prose wrapping
- semantic boundaries

と近い文化。

例えば：

```
断熱性能は、熱移動を抑制し、表面温度低下を防ぐ。
```

のように、

- clause
- phrase
- 意味単位

で改行する。

---

# 3. Markdown 本来の仕様

ここで CommonMark / Markdown 本来の仕様を整理。

通常 Markdown では：

```
AB
```

は：

> 同一段落

として扱われる。

つまり、

- 単一改行 = soft break
- 空行 = paragraph break

である。

---

# 4. Quartz の HardLineBreaks plugin

次に Quartz に存在する：

- `HardLineBreaks`

plugin の話へ。

Quartz 公式では：

> “doesn’t follow the semantics of actual Markdown”

と明記されている。

つまり Quartz 側も：

- Markdown 本来の意味論
- semantic structure

から外れていることを理解している。

---

# 5. では、なぜ実装されているのか？

ここで重要な議論。

当初は：

- 日本語は改行文化が強い
- Web 上では短行文化が強い

ために実装されているのでは、  
という仮説が出た。

しかしユーザーから：

> Quartz / Obsidian において、日本語ユーザーは少数。  
> 日本人向け実装とは考えにくい。

という重要な指摘があった。

---

# 6. 英語圏でも「Web publishing」需要が強い

その結果、議論は：

> HardLineBreaks は、  
> “英語圏でも Web publishing のために必要”

という方向へ整理された。

つまり：

- Markdown は元々 plaintext 思想
- しかし現代では publishing format 化している

という変化。

---

# 7. remark-breaks の存在

ここで：

- Next.js
- Node.js ecosystem

に存在する：

- `remark-breaks`

の話へ。

remark-breaks は：

```
AB
```

を：

```
A<br>B
```

に変換するライブラリ。

つまり：

> 単一改行を hard break 化

する。

---

# 8. remark-breaks が存在する理由

ここで議論されたのは：

> 「Markdown本来と違うのに、  
> なぜ ecosystem に存在するのか？」

という点。

整理として：

- Markdown本来は semantic plaintext
- しかし現代 Web では  
    「Enter したら改行されて欲しい」  
    需要が強い

ため、

- publishing
- readability
- mobile UX

寄りの需要を吸収するために存在している、  
と整理された。

---

# 9. semantic 派は何を守ろうとしているのか

その後：

> 「では、現在も semantic 派は何をもって  
> semantic line breaks を採用しているのか？」

という議論へ。

semantic 派は：

- diff readability
- editability
- portability
- source structure

を重視している、  
という整理を行った。

---

# 10. Git diff の話

ここでユーザーから：

> Git diff の意味がよく分からない

という話が出た。

そこから：

## semantic line breaks なし

```
断熱性能は熱移動を抑制し表面温度低下を防ぐ。
```

↓

```
断熱性能は熱移動を抑制し放熱を低減し表面温度低下を防ぐ。
```

diff：

```
- 行全体+ 行全体
```

---

## semantic line breaks あり

```
断熱性能は、熱移動を抑制し、表面温度低下を防ぐ。
```

↓

```
断熱性能は、熱移動を抑制し、放熱を低減し、表面温度低下を防ぐ。
```

diff：

```
+ 放熱を低減し、
```

のように、

> 「どこが変更されたか」

を局所化できる、  
という話になった。

---

# 11. しかし、個人運用では差が薄いのでは？

ここでユーザーから重要な指摘。

> 「他人の文章なら diff readability は重要。  
> でも自分の文章は、  
> “ここ違和感あるから書き換えよう”  
> くらいの感覚で編集する」

つまり：

- line-level diff
- clause-level diff

ではなく、

> 段落・思考単位

で編集している。

---

# 12. semantic line breaks の恩恵は

「共同編集文化」で大きい

ここで整理されたのは：

semantic line breaks の価値は、

- OSS docs
- RFC
- API docs
- チーム編集

など、

> “他人の変更を読む文化”

で最大化される。

一方、ユーザーのような：

- 個人PKM
- Obsidian
- Quartz
- GitHubは同期用途

では恩恵が減る。

---

# 13. 「publish 向けでも知識資産では？」

ここでさらに本質的な疑問へ。

ユーザーは：

> 「publish 向けでも、  
> 文章として残るなら知識資産では？」

と指摘。

また：

> Zettelkasten 的には、  
> 1トピック1カードとリンク構造が重要では？

という視点を提示。

---

# 14. semantic line breaks と

Zettelkasten は本来別

この流れで整理されたのは：

- semantic line breaks
- Git diff
- plaintext purity

は、

> “プレーンテキスト編集文化”

であり、

---

一方、

Zettelkasten の本質は：

- atomicity
- linking
- reusability
- thought connection

である。

つまり：

> semantic line breaks を採用しなくても、  
> Zettelkasten は成立する。

---

# 15. ユーザーの最終的な立場

最後にユーザーは：

> 「書くための効率性は必要。  
> しかし、自分が読めないカードは  
> Zettelkasten として利用価値が減る」

と整理した。

つまり：

- diff readability
- line-level editability

より、

> 「再読可能性」

を優先している。

---

# 16. 最終整理された方向性

ユーザーの志向として整理されたのは：

## semantic 派

重視：

- source optimization
- diff readability
- semantic structure
- editability

---

## ユーザー

重視：

- 再読可能性
- publish readability
- 思考再接続
- 読みやすいカード
- ノート間リンク構造

---

つまりユーザーは：

> 「semantic line breaks 原理主義」

ではなく、

> 「読める Zettelkasten」

を最大化したい立場。

---

# 17. 結論として見えてきた方向

議論全体を通して、ユーザーに合う方向性としては：

- HardLineBreaks ON
- 読みやすい文章
- publishability 重視
- atomic note は維持
- note linking は維持
- frontmatter / tags 整理

という方向性がかなり自然、  
という整理になった。

つまり：

> 「semantic line breaks の美しさ」

より、

> 「自分が再読できること」

を優先する方向。
